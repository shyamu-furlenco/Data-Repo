# Table: renewals

## Metadata
layer: Silver (CDC)
full_path: furlenco_silver.order_management_systems_evolve.renewals
row_count_approx: 3,987,907
refresh_cadence: continuous (CDC)

## Description
One row represents a single renewal of one rentable entity — an Item, Bundle, Plan, or CompositeItem. Multiple renewals can belong to a single `renewal_transaction_id` when multiple separate entities are renewed in a single transaction (e.g., a user renewing two bundles at once). Each row corresponds to exactly one entity — a bundle renewal creates ONE row with `entity_type=BUNDLE`, not one row per item in the bundle. The table answers questions about when products were renewed, how much was paid, what offers were applied, and what stage a renewal is in (`AWAITING_PAYMENT`, `FUTURE`, `APPLIED`, or `CANCELLED`). Key joins: `renewal_transaction_id` → `renewal_transactions.id`; `entity_id` + `entity_type` → the corresponding entity table (items, bundles, plans, composite_items). The `vertical` column (`FURLENCO_RENTAL` or `UNLMTD`) determines which product type is being renewed.

## State values
| State | Meaning |
|-------|---------|
| `NULL` | Initial builder state. No production rows exist in this state — the renewal transitions to `AWAITING_PAYMENT` immediately on creation. Defined in enum only. (0 rows in current data.) |
| `AWAITING_PAYMENT` | Renewal has been created and is waiting for payment (via `RenewalOrderedEventHandler`). All key columns (`renewal_transaction_id`, `payment_details`, `pricing_details`, `applicable_on`, `cancellable_until`) are set on entry. Transitions to `FUTURE` once payment is confirmed. The user can cancel only while today ≤ `cancellable_until` (= the entity's tenure end date; the cancellation offset is 0 days for both verticals). Once that date passes, cancellation is blocked — the renewal stays in this state until payment arrives. No auto-cancellation exists. (~0.07%) |
| `FUTURE` | Payment received and confirmed (via `RenewalPaymentCompletedEventHandler`). Renewal is scheduled to be applied on the `applicable_on` date. The product has not yet been extended — extension happens on `applicable_on`. Can still be cancelled if today ≤ `cancellable_until`. (~0.4%) |
| `APPLIED` | Terminal. Renewal has been applied to the product (via `RenewalAppliedEventHandler`). `applied_on` is set to the date the renewal was activated. The product's tenure has been extended, old VAS expired, new VAS activated, and a `CompletedTenure` record created. (~73.1%) |
| `CANCELLED` | Terminal. Renewal was cancelled — either by the user within the cancellation window (via `RenewalCancelledEventHandler`) or as the result of repeated payment failure. VAS associated with this renewal is also cancelled. `applied_on` is null. (~26.4%) |

## State groupings
| Group | Enum constant | States | Notes |
|-------|--------------|--------|-------|
| Cancellable | `CANCELLABLE_STATES` | `AWAITING_PAYMENT`, `FUTURE` | Rows where the user can still cancel. Also guarded by `cancellable_until` date — must be today or later. |
| Ignored for auto-renewal | `STATES_TO_IGNORE_FOR_AUTO_RENEWAL` | `FUTURE`, `AWAITING_PAYMENT` | Excluded when the system evaluates whether a new auto-renewal should be created for an entity. |
| Paid renewals | `hasPaidRenewals()` | `APPLIED`, `FUTURE` | Renewals where payment has been received. Used to determine if an entity already has a paid renewal on record. |

## Common queries

```sql
-- 1. Renewals currently pending application (FUTURE or AWAITING_PAYMENT)
SELECT state, COUNT(*) AS cnt
FROM furlenco_silver.order_management_systems_evolve.renewals
WHERE state IN ('FUTURE', 'AWAITING_PAYMENT')
GROUP BY state;
```

```sql
-- 2. Renewal revenue for a date range (APPLIED renewals, by date applied)
SELECT
  applied_on,
  COUNT(*) AS renewal_count,
  SUM(CAST(payment_details_payable:total AS DECIMAL(12,2))) AS total_revenue
FROM furlenco_silver.order_management_systems_evolve.renewals
WHERE state = 'APPLIED'
  AND applied_on BETWEEN '2026-01-01' AND '2026-03-31'
GROUP BY applied_on
ORDER BY applied_on;
```

```sql
-- 3. All renewals for a specific entity (e.g., BUNDLE id 301549)
SELECT id, display_id, state, applicable_on, applied_on, tenure_in_months, type
FROM furlenco_silver.order_management_systems_evolve.renewals
WHERE entity_id = 301549
  AND entity_type = 'BUNDLE'
ORDER BY created_at DESC;
```

```sql
-- 4. Cancellation rate by vertical (terminal states only)
SELECT
  vertical,
  SUM(CASE WHEN state = 'APPLIED' THEN 1 ELSE 0 END) AS applied_cnt,
  SUM(CASE WHEN state = 'CANCELLED' THEN 1 ELSE 0 END) AS cancelled_cnt,
  SUM(CASE WHEN state = 'CANCELLED' THEN 1 ELSE 0 END) * 100 /
    SUM(CASE WHEN state IN ('APPLIED', 'CANCELLED') THEN 1 ELSE 0 END) AS cancellation_pct
FROM furlenco_silver.order_management_systems_evolve.renewals
WHERE state IN ('APPLIED', 'CANCELLED')
GROUP BY vertical;
```

```sql
-- 5. Renewal breakdown by entity type and vertical
SELECT entity_type, vertical, COUNT(*) AS cnt
FROM furlenco_silver.order_management_systems_evolve.renewals
GROUP BY entity_type, vertical
ORDER BY cnt DESC;
```

## Lifecycle column changes

### On renewal creation (NULL → AWAITING_PAYMENT)
`RenewalOrderedEventHandler` creates the row. All NOT NULL columns are set at this point.

| Column | Change |
|--------|--------|
| `state` | Set to `AWAITING_PAYMENT` |
| `display_id` | Generated by `RenewalDisplayIdGenerator` (prefix `RNW` + 10-digit number) |
| `payment_details` | Set from the renewal order request |
| `pricing_details` | Set from catalog pricing at time of order |
| `offers_snapshot` | Set from applicable offers |
| `applicable_on` | Set to `entity.tenureEndDate + 1 day` |
| `cancellable_until` | Set to `entity.tenureEndDate - CANCELLATION_PERIOD_END_OFFSET` (currently 0 days for both verticals) |
| `applied_on` | Remains null |

### On payment completion (AWAITING_PAYMENT → FUTURE)
`RenewalPaymentCompletedEventHandler` confirms payment.

| Column | Change |
|--------|--------|
| `state` | Set to `FUTURE` |
| `updated_at` | Updated |

### On renewal applied (FUTURE → APPLIED)
`RenewalAppliedEventHandler` fires on `applicable_on` date.

| Column | Change |
|--------|--------|
| `state` | Set to `APPLIED` |
| `applied_on` | Set to the date of application |
| `updated_at` | Updated |

_Side effects on parent entity: old VAS expired, new VAS activated, `CompletedTenure` record created, renewal overdue cycle dates nullified._

### On cancellation (AWAITING_PAYMENT/FUTURE → CANCELLED)
`RenewalCancelledEventHandler` handles user-initiated cancellation.

| Column | Change |
|--------|--------|
| `state` | Set to `CANCELLED` |
| `updated_at` | Updated |

_Side effects on parent entity: renewal payment/pricing fields reset, VAS cancelled. A new renewal may be triggered if the renewal window is still open._

### On payment failure
`RenewalPaymentFailedEventHandler` handles failed payment attempts. The renewal row itself is **not immediately deleted or state-transitioned** — it may remain in `AWAITING_PAYMENT` and eventually become `CANCELLED`. Side effects land on the parent entity: overdue cycle dates set, VAS cancelled, penalties created.

## Caveats
- All timestamp columns (`created_at`, `updated_at`, `ingestion_timestamp`) are stored in UTC. Convert to IST: `CONVERT_TIMEZONE('UTC', 'Asia/Kolkata', col)`.
- `_rescued_data` is an Auto Loader system column — ignore for analytics.
- `catalog_plan_snapshot` and `offers_snapshot` are stored as raw JSON **strings** (not VARIANT) in Databricks. Do not use variant dot-notation (`:key`). Parse with `FROM_JSON()` or `PARSE_JSON()` instead.
- All monetary values in `payment_details_*` and `pricing_details_*` VARIANT sub-columns are **quoted decimal strings** (e.g., `"1310.64"`). Always `CAST(col AS DECIMAL(12,2))` before arithmetic.
- `applied_on` is null for ~26.9% of rows (all non-APPLIED states). Filter `WHERE applied_on IS NOT NULL` when computing revenue or applied-renewal metrics.
- `catalog_plan_snapshot` is null for ~95.3% of rows (all `FURLENCO_RENTAL` renewals). It is populated only for UNLMTD renewals where `entity_type = 'PLAN'`.
- `type = 'TENURE_EXTENSION'` (~0.2% of rows) are early renewals placed when the entity's tenure end date is more than 30 days away. Exclude from standard monthly renewal-cycle analyses.
- One `renewal_transaction_id` can appear on multiple renewal rows when multiple separate entities are renewed in a single transaction. A bundle renewal creates exactly ONE row with `entity_type=BUNDLE` — not one row per item in the bundle.
- `ATTACHMENT` is a valid value in the source `RenewableEntityType` enum but has 0 rows in production data — do not treat it as an active entity type.
- `AUTOPAY` is a valid value in the source `RenewalType` enum but has 0 rows in production data.
